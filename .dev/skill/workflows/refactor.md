# 工作流：重构

## 触发命令
`/app refactor [target]` — 如 `/app refactor skin-record-module`, `/app refactor auth-service`, `/app refactor navigation`

## 执行流程

### 第1步：上下文加载
1. 加载记忆：
   - `app-decisions.md` — 已有架构决策
   - `app-issues.md` — 相关技术债务
   - `app-learnings.md` — 相关经验
2. 读取目标代码

### 第2步：分析现状
1. 识别问题（代码异味）：
   - 重复代码
   - 过长函数/类
   - 不合理的依赖
   - 命名不清晰
   - 架构违反
2. 量化影响：
   - 影响的文件数
   - 改动的风险评估

### 第3步：制定重构方案
```
重构目标: [一句话]
原因: [为什么要重构]
方案: [具体怎么改]
影响范围: [涉及的文件]
风险: [可能引入的问题]
回滚计划: [如果出问题怎么办]
```

### 第4步：用户确认
展示方案，等待确认。

### 第5步：执行重构
1. 按照方案逐步执行
2. 每步修改后验证编译通过
3. 运行相关测试

### 第6步：更新记忆
- `app-decisions.md` — 如果是架构级重构，记录ADR
- `app-issues.md` — 关闭相关技术债务
- `app-learnings.md` — 记录重构经验

---

## 重构安全规则
- [ ] 不改变外部行为（纯内部优化）
- [ ] 每步改动可编译
- [ ] 有测试覆盖的代码先运行测试
- [ ] 大范围重构分多步进行
- [ ] 重构和功能修改不混在一起

---

## 输出格式

```
🔄 重构: [target]

━━━ 问题分析 ━━━
[识别的代码异味]

━━━ 重构方案 ━━━
[具体修改计划]

━━━ 执行结果 ━━━
修改文件: [X]个
[修改摘要]

━━━ 验证 ━━━
编译: ✅ | 测试: ✅

━━━ 记忆更新 ━━━
[已更新的记忆文件]
```
