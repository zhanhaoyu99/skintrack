# 工作流：Bug诊断修复

## 触发命令
`/app bug [description]` — 如 `/app bug 登录接口返回500`, `/app bug 图片加载白屏`

## 执行流程

### 第1步：上下文加载
1. 加载记忆文件：
   - `app-issues.md` — 检查是否是已知问题
   - `app-progress.md` — 定位相关模块
   - `app-learnings.md` — 检查类似问题是否有历史解法

### 第2步：问题分类
```
Bug ID: #[自增编号]
日期: [YYYY-MM-DD]
严重度: 🔴Critical / 🟡Major / 🟢Minor
模块: [Frontend/Backend/Shared/Database/DevOps]
描述: [用户报告的问题]
复现步骤: [1. 2. 3.]
期望行为: [应该怎样]
实际行为: [实际怎样]
```

### 第3步：诊断
1. **定位代码**：
   - 根据错误信息找到相关文件
   - 检查日志/堆栈跟踪
   - 确定问题所在层（UI/ViewModel/Service/Repository/数据库）

2. **根因分析**：
   - 是逻辑错误？
   - 是空指针/类型错误？
   - 是并发/竞态条件？
   - 是配置/环境问题？
   - 是第三方库问题？

3. **记录已尝试方案**（避免重复）

### 第4步：修复
1. 编写修复代码
2. 验证修复有效
3. 检查修复是否引入新问题
4. 如果需要，补充回归测试

### 第5步：更新记忆
1. 更新 `app-issues.md`：
   - 如果是新问题 → 添加到 OPEN 或 RESOLVED
   - 如果是已知问题解决 → 移到 RESOLVED，记录解决方案
2. 如果有教训 → 更新 `app-learnings.md`
3. 如果发现技术债务 → 添加到 `app-issues.md` 技术债务表

---

## Issue记录格式
```markdown
### #[ID] [严重度] [标题]
- **日期**: YYYY-MM-DD
- **模块**: Frontend/Backend/Shared
- **描述**: [问题描述]
- **复现**: [步骤]
- **已尝试**: [尝试过的方案]
- **根因**: [根本原因]
- **解决方案**: [最终修复]
- **教训**: [避免再犯的经验]
```

---

## 输出格式

```
🐛 Bug修复: [标题]

━━━ 诊断 ━━━
根因: [根本原因分析]
影响范围: [受影响的功能]

━━━ 修复 ━━━
[修改的文件 + 修改内容]

━━━ 验证 ━━━
[验证修复有效的步骤]

━━━ 记忆更新 ━━━
app-issues.md: [已记录 #ID]
app-learnings.md: [已记录教训（如有）]
```

---

## 注意事项
- 先诊断再修复，不要盲目猜测
- 如果根因不明确，标记为🟡继续调查
- 修复后要验证，不要假设修复有效
- 记录教训，同类问题不犯第二次
