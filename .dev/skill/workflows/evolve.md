# 工作流：自我进化

## 触发命令
`/app evolve` — 全面自我审视和进化

## 触发建议
- 每完成一个里程碑后
- 积累了大量开发经验后
- 技术栈有重大更新时
- 多次遇到类似问题时

---

## 执行流程

### 第1步：全量上下文加载

**记忆文件（5个）：**
```
app-progress.md
app-issues.md
app-decisions.md
app-strategy.md
app-learnings.md
```

**知识库文件（10个）：**
```
knowledge/kmp-architecture.md
knowledge/compose-multiplatform.md
knowledge/spring-boot-patterns.md
knowledge/api-design.md
knowledge/database-patterns.md
knowledge/testing-strategy.md
knowledge/security-checklist.md
knowledge/code-style.md
knowledge/devops-deploy.md
knowledge/skin-domain.md
```

### 第2步：四维度分析

**维度1：开发效率**
- 哪些工作流高效？哪些低效？
- 哪些知识缺口导致了卡点？
- 代码质量趋势如何？

**维度2：架构健康**
- 模块划分是否合理？
- 依赖关系是否清晰？
- 有没有反模式累积？

**维度3：问题模式**
- 反复出现的问题类型
- 技术债务积累趋势
- 根因分析是否到位

**维度4：知识准确性**
- knowledge/ 中哪些建议被实践验证有效？
- 哪些建议被证明过时或不适用？
- 有没有实践中发现但知识库中缺失的经验？

### 第3步：优化识别

**🔴 紧急更新（知识过时/被证明错误）：**
```
文件：[文件名]
问题：[当前内容]
原因：[为什么需要更新]
建议：[具体修改方案]
```

**🟡 建议更新（可优化）：**
```
文件：[文件名]
问题：[当前内容]
原因：[优化理由]
建议：[具体修改方案]
```

**🟢 已验证有效（可强化）：**
```
文件：[文件名]
内容：[有效的部分]
证据：[实践支撑]
```

### 第4步：进化方案

```
━━━ 进化方案 ━━━

总计需要更新的文件：[X] 个

文件1：[文件名]
├── 当前状态：[问题描述]
├── 进化方向：[具体修改]
├── 依据：[数据/经验支撑]
└── 优先级：[高/中/低]
```

### 第5步：用户确认

展示完整方案，等待确认：
```
以上是本次进化方案，共涉及 [X] 个文件的 [Y] 处修改。
确认执行？(Y/N)
```

### 第6步：批量执行

确认后使用 Edit 工具逐一更新：
1. 更新文件时间戳
2. 执行具体修改
3. 保持文件结构

### 第7步：进化日志

在 `app-learnings.md` 中记录：
```markdown
## [日期] 系统进化

**触发原因：** [里程碑完成/定期/问题驱动]
**分析范围：** [X]个记忆文件 + [Y]个知识文件

**修改汇总：**
| 文件 | 修改类型 | 内容摘要 |
|------|---------|---------|
| [文件名] | 更新/新增/删除 | [简述] |

**预期影响：**
- [对后续开发的影响]

**验证计划：**
- [如何验证进化效果]
```

---

## 安全机制

### 审批机制
- 所有修改必须经过用户确认
- 批量修改前展示完整方案

### 可回滚
- 旧信息标注"过时"而非删除
- app-learnings.md 记录所有变更
- 如果进化后出问题，下次进化可以回退

### 数据驱动
- 不基于猜测修改知识库
- 每个修改必须有实践依据
- 不确定的信息标注为"待验证"
