# 工作流：Sprint规划

## 触发命令
`/app sprint` — 规划下一批开发工作

## 执行流程

### 第1步：全量状态加载
加载所有记忆文件：
```
app-progress.md    — 当前进度
app-issues.md      — 待解决问题
app-decisions.md   — 架构决策
app-strategy.md    — 项目策略
app-learnings.md   — 经验教训
```

### 第2步：当前状态分析
1. 当前里程碑完成度
2. 阻塞问题清单
3. 技术债务积累情况
4. 上一批工作的回顾

### 第3步：优先级排序
按以下优先级排序待做事项：
1. 🔴 阻塞问题（BLOCKING）→ 必须先解决
2. 🟡 当前里程碑未完成任务 → 按依赖关系排序
3. 🟢 技术债务（如积累过多）→ 穿插处理
4. 🔵 下一里程碑准备 → 提前设计

### 第4步：Sprint定义
```
Sprint #[N]
目标: [一句话目标]
范围: [里程碑/模块]

━━━ 任务清单 ━━━
| # | 任务 | 类型 | 命令 | 优先级 | 依赖 |
|---|------|------|------|--------|------|
| 1 | 修复登录bug | bug | /app bug ... | 🔴 | — |
| 2 | 完成记录列表页 | page | /app page ... | 🟡 | — |
| 3 | 创建记录详情API | api | /app api ... | 🟡 | #2 |
| 4 | 补充auth测试 | test | /app test ... | 🟢 | — |

━━━ 完成标准 ━━━
- [ ] 所有🔴任务完成
- [ ] 页面可正常使用
- [ ] API测试通过
- [ ] 无新增BLOCKING问题

━━━ 注意事项 ━━━
[基于learnings的提醒]
```

### 第5步：用户确认
展示Sprint计划，等待用户确认或调整。

### 第6步：更新记忆
确认后更新 `app-progress.md`：
- 当前工作描述
- 下一步计划

---

## 输出格式

```
📅 Sprint规划

━━━ 当前状态 ━━━
里程碑: M[X] — [名称] ([X]%)
阻塞: [X]个
技术债务: [X]个

━━━ Sprint #[N] 计划 ━━━
目标: [一句话]

任务列表:
[按优先级排列的任务表格]

预计交付:
[完成后的状态描述]

确认开始？(Y/N)
```

---

## 注意事项
- Sprint不要规划过多任务，保持可完成
- 阻塞问题优先于新功能
- 技术债务不能无限积累，每个Sprint至少处理1个
- 保持里程碑聚焦，不跳跃开发
